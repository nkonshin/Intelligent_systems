## 1. Постановка задачи

### 1.1 Описание предметной области

Системы городского велопроката (bike-sharing) работают по модели краткосрочного проката: пользователь берёт велосипед на одной станции и возвращает на другой. Ключевая проблема оператора — дисбаланс велосипедов между станциями: в утренний час пик велосипеды "уезжают" из жилых районов в деловые, вечером — обратно. Если заранее знать ожидаемый спрос, можно перераспределять велосипеды до начала пикового часа.

### 1.2 Формулировка задачи

Задача **регрессии**: по набору признаков (время суток, день недели, погодные условия) предсказать количество велопрокатов за один час. Целевая переменная — `cnt` (count), непрерывная числовая величина.

### 1.3 Целевые пользователи

- Операторы велопрокатных систем — для планирования перераспределения велосипедов между станциями
- Городские транспортные департаменты — для развития инфраструктуры
- Аналитики урбанистических данных

### 1.4 Гуманитарный / социальный аспект

Повышение доступности и удобства городского велопроката, снижение нагрузки на общественный транспорт, популяризация экологичного транспорта и улучшение пользовательского опыта жителей и туристов.

## 2. Данные

### 2.1 Источник

Датасет **Bike Sharing Dataset** из репозитория UCI Machine Learning Repository. Содержит почасовые данные системы Capital Bikeshare (Вашингтон, США) за 2011-2012 годы, обогащённые метеорологическими данными.

- Источник: https://archive.ics.uci.edu/dataset/275/bike+sharing+dataset
- Файл: `hour.csv`
- Объём: 17,379 записей, 17 столбцов
- Пропуски: отсутствуют

### 2.2 Описание признаков

| Признак | Тип | Описание | Диапазон |
|---------|-----|----------|----------|
| `season` | Категориальный | Сезон | 1=весна, 2=лето, 3=осень, 4=зима |
| `yr` | Категориальный | Год | 0=2011, 1=2012 |
| `mnth` | Категориальный | Месяц | 1-12 |
| `hr` | Категориальный | Час суток | 0-23 |
| `holiday` | Категориальный | Праздничный день | 0/1 |
| `weekday` | Категориальный | День недели | 0=Вс, 6=Сб |
| `workingday` | Категориальный | Рабочий день | 0/1 |
| `weathersit` | Категориальный | Погодная ситуация | 1=ясно, 2=туман, 3=дождь, 4=шторм |
| `temp` | Числовой | Температура (нормализованная) | 0-1 (0=-8C, 1=+39C) |
| `atemp` | Числовой | Ощущаемая температура | 0-1 (0=-16C, 1=+50C) |
| `hum` | Числовой | Относительная влажность | 0-1 |
| `windspeed` | Числовой | Скорость ветра (нормализованная) | 0-1 (0=0, 1=67 км/ч) |

**Целевая переменная:** `cnt` — общее количество прокатов за час (1-977, среднее 189, медиана 142).

### 2.3 Удалённые столбцы

| Столбец | Причина удаления |
|---------|-----------------|
| `instant` | Порядковый номер строки, не несёт информации |
| `dteday` | Дата в строковом формате; информация уже содержится в `yr`, `mnth`, `hr`, `weekday` |
| `casual` | Количество незарегистрированных прокатов — утечка данных (data leakage) |
| `registered` | Количество зарегистрированных прокатов — утечка данных |

`casual + registered = cnt`, поэтому использование этих столбцов как признаков привело бы к тривиальному решению, не применимому на практике (эти значения неизвестны до момента, когда `cnt` уже известен).

## 3. Подготовка данных

### 3.1 Разделение выборок

Данные разделены на обучающую (80%) и тестовую (20%) выборки с фиксированным `random_state=42` для воспроизводимости:
- Обучающая выборка: 13,903 записи
- Тестовая выборка: 3,476 записей

### 3.2 Предобработка признаков

Использован `ColumnTransformer` для раздельной обработки двух типов признаков:

**Категориальные признаки (8 штук) — OneHotEncoder:**
- Параметр `drop="first"`: удаление первой категории для предотвращения мультиколлинеарности (если все dummy-переменные равны 0, модель знает, что это первая категория)
- Параметр `handle_unknown="ignore"`: при появлении неизвестной категории подставляется нулевой вектор (вместо ошибки)

**Числовые признаки (4 штуки) — StandardScaler:**
- Приведение к стандартному масштабу: z = (x - mean) / std

### 3.3 Pipeline

Предобработка и модель объединены в `sklearn.pipeline.Pipeline`. Это обеспечивает:
1. Корректность кросс-валидации (scaler обучается только на train-фолде)
2. Атомарность сохранения (модель и предобработка в одном `.pkl` файле)
3. Простоту использования (один вызов `predict()` для новых данных)

## 4. Обучение моделей

### 4.1 Выбор алгоритмов

Выбраны три принципиально разных подхода:

**1. Baseline — собственная реализация (HourlyMeanRegressor):**
- Простое правило на одном признаке: предсказание = среднее количество прокатов за конкретный час суток
- Игнорирует все остальные признаки (погоду, день недели и т.д.)
- Реализован как класс с методами `fit()` и `predict()`, наследуется от `BaseEstimator` и `RegressorMixin`
- Используется как точка отсчёта для оценки сложных моделей

**2. RandomForestRegressor (Bagging):**
- 200 деревьев решений, обучаемых параллельно на случайных подвыборках данных и признаков
- Итоговое предсказание — среднее всех деревьев
- Снижает дисперсию (variance) предсказаний

**3. GradientBoostingRegressor (Boosting):**
- 200 деревьев, обучаемых последовательно; каждое следующее исправляет ошибки предыдущих
- `max_depth=5` — неглубокие деревья ("слабые ученики")
- Снижает смещение (bias) предсказаний

### 4.2 Кросс-валидация

Для каждой модели проведена 5-fold кросс-валидация на обучающей выборке:

| Модель | CV MAE | CV RMSE |
|--------|--------|---------|
| Baseline (среднее за час) | 88.54 +/- 0.99 | 128.69 +/- 1.33 |
| RandomForest | 35.54 +/- 0.83 | 55.72 +/- 1.25 |
| GradientBoosting | 36.63 +/- 0.98 | 54.03 +/- 1.40 |

Низкое стандартное отклонение (0.83-1.33) свидетельствует о стабильности всех моделей.

### 4.3 Оценка на тестовой выборке

| Модель | MAE | RMSE | R2 |
|--------|-----|------|----|
| Baseline (среднее за час) | 87.34 | 126.83 | 0.4920 |
| **RandomForest** | **32.67** | **51.48** | **0.9163** |
| GradientBoosting | 34.86 | 51.92 | 0.9149 |

**Лучшая модель: RandomForest** с MAE = 32.67 прокатов и R2 = 0.9163.

Сложные модели снижают MAE в 2.7 раза по сравнению с baseline (87.34 → 32.67), что демонстрирует важность учёта дополнительных признаков (погода, день недели, сезон) помимо часа суток.

### 4.4 Интерпретация метрик

- **MAE = 32.67**: модель в среднем ошибается на 33 проката. При среднем спросе 189 — это ошибка ~17%.
- **RMSE = 51.48**: выше MAE, что указывает на наличие отдельных записей с крупными ошибками (часы пик).
- **R2 = 0.9163**: модель объясняет 91.6% дисперсии спроса. Оставшиеся 8.4% — факторы, отсутствующие в данных (мероприятия, аварии, временные закрытия станций).

## 5. Анализ результатов

### 5.1 Визуализация предсказаний

График "предсказанное vs реальное" показывает, что точки группируются вдоль линии y=x, что подтверждает высокое качество модели. Наибольший разброс наблюдается при высоких значениях спроса (>500 прокатов/час).

### 5.2 Анализ ошибок

Распределение остатков (residuals) близко к нормальному с центром около нуля — систематического смещения нет.

Часы с наибольшими ошибками:
| Час | Средняя ошибка |
|-----|---------------|
| 17:00 | 63 проката |
| 18:00 | 57 прокатов |
| 19:00 | 47 прокатов |

Ошибки концентрируются в вечерний час пик (17-19), когда спрос максимально волатилен.

### 5.3 Важность признаков

Топ-5 наиболее значимых признаков (по feature_importances_ RandomForest):
1. **hr** (час суток) — доминирующий фактор, спрос имеет характерный двугорбый профиль (пики в 8:00 и 17:00)
2. **temp** (температура) — тёплая погода стимулирует прокат
3. **yr** (год) — в 2012 спрос выше, чем в 2011 (рост популярности сервиса)
4. **hum** (влажность) — высокая влажность снижает спрос
5. **workingday** (рабочий день) — разный профиль спроса в будни и выходные

### 6.2 Взаимодействие компонентов

```
train.py ──сохраняет──> models/best_model.pkl
                              │
predict.py ───загружает───────┘
     │
     ├──── app.py (Gradio-интерфейс вызывает predict())
     └──── tests/test_basic.py (тесты вызывают predict())
```

**predict.py** — центральный модуль. Функция `predict(...)` принимает 12 параметров и возвращает число (прогноз прокатов). Модель загружается лениво (один раз при первом вызове) через паттерн singleton.

### 6.3 Веб-интерфейс

Реализован на фреймворке **Gradio**. Интерфейс включает:
- Выпадающие списки для категориальных признаков (сезон, погода, день недели) с человекочитаемыми подписями на русском языке
- Слайдеры для числовых признаков (температура, влажность, ветер)
- Чекбоксы для бинарных признаков (праздник, рабочий день)
- Три готовых примера для быстрой демонстрации

Результат включает числовое значение прогноза и текстовую интерпретацию уровня спроса (низкий / средний / высокий / очень высокий).

## 9. Выводы

1. Построена модель прогнозирования спроса на велопрокат с R2 = 0.916 — модель объясняет 91.6% дисперсии спроса.

2. Средняя абсолютная ошибка (MAE) составляет 33 проката при среднем спросе 189 — относительная ошибка ~17%.

3. Сравнили три принципиально разных подхода: собственная Baseline-реализация (среднее за час), RandomForest (bagging) и GradientBoosting (boosting). RandomForest показал лучшие результаты — MAE в 2.7 раза ниже, чем у Baseline.

4. Наиболее значимые факторы спроса: час суток, температура и год (тренд роста популярности сервиса).

5. Наибольшие ошибки модели приходятся на вечерний час пик (17-19), когда спрос максимально нестабилен.

6. Создан рабочий веб-прототип на Gradio, позволяющий делать прогнозы в интерактивном режиме.

7. Реализовано покрытие автотестами (pytest): проверка функции предсказания, формата ответа и запуска интерфейса. Все 3 теста проходят.
