# Практическое занятие 1: Базовый конвейер ML — от данных до модели

## Суть задания

Построить и сравнить три модели машинного обучения для прогнозирования оттока клиентов телеком-компании. Датасет `telecom_churn.csv` — бинарная классификация: ушёл клиент (1) или остался (0).

---

⏺ Лаба 1 — базовый конвейер ML (отток клиентов телеком):

  1. Загружаем telecom_churn.csv — смотрим размер, типы столбцов, пропуски
  2. Изучаем целевую переменную Churn — сколько ушло/осталось (~15%/85%)
  3. Отделяем X от y, делим 75/25
  4. Создаём ColumnTransformer — StandardScaler для числовых, OneHotEncoder для категориальных
  5. Обучаем три модели через Pipeline: LogisticRegression, DecisionTree, KNN — для каждой считаем accuracy
  6. Сравниваем accuracy в таблице и на графике — выбираем лучшую
  7. Для лучшей строим confusion matrix — смотрим FP (ложные тревоги) и FN (пропущенные уходы), объясняем что FN для бизнеса хуже

## Теоретическая база

### Бинарная классификация

Задача, где целевая переменная принимает два значения (0/1, да/нет). В нашем случае — отток клиентов (Churn). Модель учится на исторических данных и предсказывает, уйдёт ли клиент.

### Признаки (features)

Входные данные модели. Бывают:
- **Числовые** (numerical) — непрерывные значения: длительность звонков, сумма счёта, количество вызовов. Можно сравнивать, считать среднее.
- **Категориальные** (categorical) — дискретные категории: штат, тарифный план (yes/no), международный роуминг. Нельзя напрямую подать в модель — нужно закодировать в числа.

### OneHotEncoder

Кодирует категориальные признаки в бинарные столбцы. Например, столбец "State" с 50 штатами превращается в 50 столбцов (0 или 1). Признак "International plan" (yes/no) -> два столбца: `International plan_yes` и `International plan_no`.

Параметр `handle_unknown="ignore"` — если в тесте встретится категория, которой не было в train, она просто получит нули (а не ошибку).

### StandardScaler

Приводит числовые признаки к стандартному масштабу: z = (x - mean) / std. После этого среднее = 0, стандартное отклонение = 1. Нужно для моделей, чувствительных к масштабу (LogisticRegression, KNN). Без масштабирования признак "Total day minutes" (0-350) будет доминировать над "Number vmail messages" (0-50).

### ColumnTransformer

Позволяет применять разные преобразования к разным столбцам. В нашем случае: StandardScaler к числовым, OneHotEncoder к категориальным. Без него пришлось бы вручную разделять DataFrame, преобразовывать и склеивать обратно.

### Pipeline

Цепочка шагов: предобработка -> модель. Pipeline гарантирует:
1. Scaler и Encoder учатся (fit) только на train
2. К тесту применяется только transform (без утечки данных)
3. Весь процесс в одном вызове .fit() и .predict()

### Три модели

**LogisticRegression** — линейная модель. Ищет гиперплоскость, разделяющую классы. Быстрая, интерпретируемая, хорошо работает когда классы линейно разделимы. `max_iter=1000` — увеличиваем лимит итераций, чтобы алгоритм точно сошёлся.

**DecisionTreeClassifier** — дерево решений. Последовательно разбивает данные по порогам признаков ("если Total day minutes > 200 и..."). Легко интерпретируется, но склонно к переобучению (выучивает шум). Не требует масштабирования.

**KNeighborsClassifier** — метод k ближайших соседей. Для нового объекта находит k ближайших точек в обучающей выборке и предсказывает класс большинства. По умолчанию k=5. Требует масштабирования (иначе расстояния искажены). Медленный на больших данных (хранит всю выборку).

### Accuracy

Доля правильных ответов: (TP + TN) / (TP + TN + FP + FN). Простая метрика, но при дисбалансе классов (14% ушедших) модель может просто всегда предсказывать "остался" и получить 86% accuracy.

### Confusion Matrix (матрица ошибок)

Для бинарной классификации — таблица 2x2:

|  | Предсказано: Остался | Предсказано: Ушёл |
|--|--|--|
| Реально: Остался | TN (True Negative) | FP (False Positive) |
| Реально: Ушёл | FN (False Negative) | TP (True Positive) |

- **FP (ложная тревога)** — лояльного клиента назвали ушедшим. Последствие: потратим ресурсы на удержание, но клиент и так бы остался.
- **FN (пропуск)** — ушедшего клиента назвали лояльным. Последствие: клиент уходит, мы ничего не предприняли. Для бизнеса это хуже.

```python
preprocessor = ColumnTransformer(transformers=[
    ("num", StandardScaler(), num_features),
    # Шаг "num": к числовым признакам применяем StandardScaler.
    # Каждый признак → среднее=0, std=1.

    ("cat", OneHotEncoder(handle_unknown="ignore", sparse_output=False), cat_features),
    # Шаг "cat": к категориальным — OneHotEncoder.
    # handle_unknown="ignore" — неизвестные категории в тесте → вектор из нулей.
    # sparse_output=False — возвращаем обычный массив, а не разреженную матрицу.
])
# ColumnTransformer применяет разные трансформеры к разным столбцам.
# На выходе: все признаки числовые, масштабированные, готовые для моделей.
```

### Раздел 3: Обучение моделей (строки 87-112)

```python
models = {
    "Logistic Regression": LogisticRegression(max_iter=1000, random_state=42),
    # Линейная модель. max_iter=1000 — достаточно итераций для сходимости.

    "Decision Tree": DecisionTreeClassifier(random_state=42),
    # Дерево решений. По умолчанию без ограничений — может переобучиться.

    "K-Nearest Neighbors": KNeighborsClassifier(),
    # KNN, по умолчанию k=5. Не имеет random_state (детерминирован при одинаковых данных).
}
```

```python
for name, model in models.items():
    pipeline = Pipeline(steps=[
        ("preprocessor", preprocessor),   # Шаг 1: предобработка
        ("classifier", model),            # Шаг 2: модель
    ])
    # Pipeline связывает предобработку и модель.
    # При fit: preprocessor.fit_transform(X_train), затем model.fit(результат, y_train).
    # При predict: preprocessor.transform(X_test), затем model.predict(результат).

    pipeline.fit(X_train, y_train)
    # Обучаем весь Pipeline: scaler+encoder учатся на train, модель обучается на результате.

    y_pred = pipeline.predict(X_test)
    # Предсказываем на тесте. Pipeline сам применит transform к X_test.

    acc = accuracy_score(y_test, y_pred)
    # Accuracy = правильных / всего.

    results[name] = {"accuracy": acc, "pipeline": pipeline, "y_pred": y_pred}
    # Сохраняем результаты для дальнейшего сравнения.
```

## Вопросы

1. **Зачем ColumnTransformer?** — Чтобы применить разные преобразования к разным типам столбцов. Числовые масштабируем (StandardScaler), категориальные кодируем (OneHotEncoder). Без него пришлось бы вручную разделять и склеивать данные.

2. **Зачем Pipeline?** — Объединяет предобработку и модель. Гарантирует, что scaler и encoder учатся только на train (нет утечки данных). Удобно: один вызов fit/predict вместо нескольких.

3. **Почему LR и KNN требуют масштабирования, а DT нет?** — LR использует градиентный спуск и чувствительна к масштабу. KNN считает расстояния — без масштабирования признак с большими числами доминирует. DT работает с порогами ("если x > 200") — масштаб не влияет.

4. **Что такое OneHotEncoder и зачем handle_unknown="ignore"?** — Превращает категории в бинарные столбцы. `handle_unknown="ignore"` — если в тесте встретится категория, не виденная в train, она получит нули вместо ошибки.

5. **Что такое FP и FN, что хуже для бизнеса?** — FP: лояльного назвали ушедшим (лишние траты на удержание). FN: ушедшего не заметили (потерян клиент). Для бизнеса FN хуже — клиент уходит без реакции.

6. **Почему accuracy может быть обманчивой?** — При 85% лояльных клиентов модель может всегда предсказывать "остался" и получить 85% accuracy. Но при этом recall для класса "ушёл" будет 0% — ни одного ушедшего не найдёт.

7. **Зачем random_state=42?** — Фиксирует генератор случайных чисел. Результаты воспроизводимы: при повторном запуске получим те же сплиты и те же модели. 42 — просто общепринятое число (из "Автостопом по галактике").

8. **Как работает KNeighborsClassifier?** — Для нового объекта находит k=5 ближайших точек в обучающей выборке (по евклидову расстоянию). Предсказывает класс, который встречается чаще среди этих соседей. Не строит модель — просто запоминает данные (lazy learner).
