## Теоретическая база

### Дисбаланс классов

В нашем датасете ~85% лояльных и ~15% ушедших. Модель может "лениться" и всегда предсказывать "остался" — accuracy будет 85%, но ни одного ушедшего клиента не найдёт. Для борьбы с дисбалансом:

- **class_weight='balanced'** — увеличивает штраф за ошибки на редком классе. Веса обратно пропорциональны частоте: w_i = n / (k * n_i). Класс "ушёл" (15%) получает вес ~3.3, "остался" (85%) — ~0.6.
- **SMOTE** — генерирует синтетические объекты редкого класса (интерполяция между ближайшими соседями). Только на train.

### False Positive vs False Negative

- **FP** (ложная тревога): лояльного клиента приняли за ушедшего. Последствие: потратим бюджет на удержание без необходимости.
- **FN** (пропуск): ушедшего клиента не заметили. Последствие: клиент уходит, мы теряем выручку.

Для бизнеса **FN критичнее** — потеря клиента дороже, чем лишний звонок лояльному. Поэтому оптимизируем **Recall** (полноту) класса "ушёл".

### Расширенные метрики

- **Precision** — из всех предсказаний "ушёл", сколько действительно ушли. Высокий precision = мало ложных тревог.
- **Recall** — из всех реально ушедших, сколько модель нашла. Высокий recall = мало пропусков.
- **F1-score** — гармоническое среднее precision и recall. F1 = 2PR / (P+R).
- **F1-macro** — среднее F1 по обоим классам. Основная метрика оптимизации.

### ROC-кривая и AUC-ROC

**ROC-кривая** (Receiver Operating Characteristic) — график зависимости TPR (True Positive Rate = Recall) от FPR (False Positive Rate) при разных порогах классификации.

Модель выдаёт вероятность (0.0-1.0). По умолчанию порог = 0.5: если p > 0.5, предсказываем "ушёл". ROC-кривая показывает, что произойдёт при сдвиге порога.

**AUC-ROC** (Area Under Curve) — площадь под ROC-кривой (от 0 до 1):
- 0.5 — случайное гадание (диагональ)
- 0.7-0.8 — приемлемо
- 0.8-0.9 — хорошо
- 0.9+ — отлично

AUC не зависит от порога — оценивает "способность модели ранжировать" (отделять классы).

### GridSearchCV

Полный перебор комбинаций гиперпараметров с кросс-валидацией:
1. Задаём сетку: какие параметры и какие значения перебирать
2. Для каждой комбинации — k-fold кросс-валидация (у нас k=5)
3. Для каждого фолда: обучение на k-1 частях, проверка на оставшейся
4. Среднее качество по фолдам — оценка комбинации
5. Выбирается лучшая комбинация

### Важность признаков (Feature Importance)

- **DecisionTree/RandomForest**: `feature_importances_` — среднее уменьшение примеси (Gini impurity) по всем деревьям. Показывает, какие признаки чаще всего и сильнее всего помогают разделять классы.
- **LogisticRegression**: `coef_` — веса признаков. Абсолютное значение |coef| показывает силу влияния. Знак — направление (+ → увеличивает вероятность класса 1).

```python
from sklearn.metrics import precision_score, recall_score, f1_score
# Дополнительные метрики для таблицы.

def metrics_row(name, y_true, y_pred, y_prob_col=None):
    row = {
        "Модель":    name,
        "Accuracy":  f"{accuracy_score(y_true, y_pred):.4f}",
        "Precision": f"{precision_score(y_true, y_pred, zero_division=0):.4f}",
        # precision_score по умолчанию для класса 1 (positive).
        # zero_division=0 — если делитель 0, вернуть 0 вместо ошибки.

        "Recall":    f"{recall_score(y_true, y_pred, zero_division=0):.4f}",
        # Recall для класса 1 (ушедших).

        "F1-macro":  f"{f1_score(y_true, y_pred, average='macro', zero_division=0):.4f}",
        # F1-macro — среднее F1 по обоим классам.
    }
    if y_prob_col is not None:
        fpr_, tpr_, _ = roc_curve(y_true, y_prob_col)
        row["AUC-ROC"] = f"{auc(fpr_, tpr_):.4f}"
    else:
        row["AUC-ROC"] = "—"
    return row
# Функция формирует строку таблицы с метриками для одной модели.
```

## Вопросы

1. **Какая проблема с базовой моделью?** — Дисбаланс: 85% лояльных, 15% ушедших. Модель хорошо предсказывает "остался", но плохо находит ушедших (низкий recall класса 1).

2. **Что такое class_weight='balanced' и как помогает?** — Увеличивает штраф за ошибки на редком классе. Формула: w = n / (k * n_i). Класс "ушёл" получает вес ~3.3 — модель "старается" его не пропускать. Recall растёт, но precision может упасть (больше ложных тревог).

3. **Зачем SMOTE, если есть class_weight?** — SMOTE создаёт реальные синтетические точки в пространстве признаков, class_weight просто меняет функцию потерь. SMOTE может давать лучшие результаты, когда классы перекрываются. В нашем коде SMOTE применён для демонстрации, но основная модель использует class_weight.

4. **Что такое ROC-кривая и AUC?** — ROC показывает зависимость True Positive Rate от False Positive Rate при разных порогах классификации. AUC — площадь под ней. AUC=0.5 — случайное гадание, AUC=1.0 — идеальная модель. AUC оценивает способность модели ранжировать объекты (отделять классы), независимо от порога.

5. **Что делает GridSearchCV?** — Перебирает все комбинации гиперпараметров. Для каждой — 5-fold кросс-валидация. Выбирает комбинацию с лучшим средним f1_macro. Это автоматизированный подбор параметров вместо ручного.

6. **Зачем Pipeline в GridSearchCV?** — Чтобы предобработка (scaler, encoder) была частью кросс-валидации. Без Pipeline scaler бы "видел" валидационный фолд — утечка данных. Pipeline гарантирует: fit только на train-фолде, transform на val-фолде.

7. **Что такое C в LogisticRegression?** — Обратная сила регуляризации. C=0.01 — сильная регуляризация (простая модель, меньше переобучение). C=10 — слабая регуляризация (сложная модель, может переобучиться). GridSearch находит оптимальный баланс.

8. **Что показывает feature_importances_?** — Вклад каждого признака в предсказания. Для деревьев — среднее уменьшение примеси Джини. Помогает понять, какие факторы влияют на отток: например, "Total day charge" или "Customer service calls".

9. **Почему F1-macro, а не accuracy?** — При дисбалансе accuracy завышается за счёт мажорного класса. F1-macro усредняет F1 по классам — оба класса одинаково важны. Если recall класса "ушёл" = 0, F1-macro будет низким, даже при accuracy = 85%.

10. **Что такое refit в GridSearchCV?** — По умолчанию True. После нахождения лучших параметров GridSearchCV автоматически переобучает модель на ВСЕХ тренировочных данных (не только на фолдах). Поэтому `best_estimator_` — уже готовая к использованию модель.
